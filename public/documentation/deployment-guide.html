1. Prerequisites & System Requirements

Required Software

Software Required Version How to Check
Node.js 20.x (exactly) node --version
Yarn 1.22.22 yarn --version
Git 2.x or higher git --version
nvm (recommended) Latest nvm --version
Critical: Node.js Version

This project requires Node.js 20.x. Using Node.js 18.x or 22.x will cause build failures. The exact version is specified in the .nvmrc file.

Installing Node.js with nvm (Recommended)

Using nvm ensures you have the correct Node.js version:

# Install nvm (if not already installed)
# macOS/Linux:
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.35.3/install.sh | bash

# Windows:
# Download and install from https://github.com/coreybutler/nvm-windows

# Restart your terminal, then:
# Install and use the correct Node.js version
nvm install 20
nvm use 20

# Verify installation
node --version  # Should show v20.x.x

Installing Yarn

# Install Yarn globally
npm install -g yarn@1.22.22

# Verify installation
yarn --version  # Should show 1.22.22
Do NOT Use npm for This Project!

This project uses Yarn exclusively. Mixing npm and Yarn will cause dependency conflicts. The package.json specifies packageManager: "yarn@1.22.22".

Required Environment Variables

The following environment variables must be configured:

Variable Purpose Required?
NUXT_PUBLIC_API_BASE_URL Strapi API endpoint Yes
NUXT_PUBLIC_BASE_URL Site base URL Yes
NUXT_MANUAL_ROUTES Routes for static generation Yes
PLAUSIBLE_DOMAIN Analytics domain No
PLAUSIBLE_API_HOST Analytics API host No

Operating System Compatibility

  • macOS: ✅ Fully supported
  • Linux: ✅ Fully supported
  • Windows: ✅ Supported via WSL2 (Windows Subsystem for Linux)
Windows Users:

For best performance on Windows, use WSL2 (Windows Subsystem for Linux). Store the project in the WSL2 file system (/home/username/) not the Windows file system (/mnt/c/).

2. Local Development Deployment

Step-by-Step Setup from Scratch

Step 1: Clone the Repository

# Clone the repository
git clone https://github.com/ICJIA/icjia-infonet-nuxt4.git

# Navigate to the project directory
cd icjia-infonet-nuxt4

Step 2: Set Up Node.js Version

# Use the correct Node.js version (reads from .nvmrc)
nvm use

# If the version isn't installed, nvm will prompt you to install it:
nvm install 20
nvm use 20

Step 3: Configure Environment Variables

# Copy the sample environment file
cp .env.sample .env

# Edit the .env file with your preferred editor
# For local development, the default values should work:
nano .env  # or vim, code, etc.

Your .env file should contain:

NUXT_PUBLIC_API_BASE_URL=https://infonet.icjia-api.cloud
NUXT_PUBLIC_BASE_URL=https://infonet.icjia.dev
NUXT_MANUAL_ROUTES=["/search", "/translate", "/faqs", "/news","/data-and-publications"]
PLAUSIBLE_DOMAIN=infonet.illinois.gov
PLAUSIBLE_API_HOST=https://analytics.icjia-api.cloud/js/plausible.js
NUXT_BASE_URL=/
NUXT_ACTIVE_SECTIONS=['root', 'page', 'meetings', 'faqs']

Step 4: Install Dependencies

# Install all project dependencies
yarn install

# This will:
# 1. Download all packages from package.json
# 2. Create/update yarn.lock
# 3. Run postinstall script (nuxt prepare)
First Install Takes Time:

The first yarn install can take 2-5 minutes depending on your internet connection. Subsequent installs will be faster due to Yarn's cache.

Step 5: Run the Development Server

# Start the development server
yarn dev

# This command does the following:
# 1. Kills any process on port 8000
# 2. Clears Nuxt cache (.nuxt and .output directories)
# 3. Runs creator scripts to fetch content from Strapi
# 4. Starts Nuxt dev server on http://localhost:8000
# 5. Opens browser automatically
Success!

If everything is working, you should see:

  • Creator scripts running and generating content
  • Nuxt dev server starting
  • Browser opening to http://localhost:8000
  • The InfoNet homepage loading

Building and Previewing Locally

Before deploying to production, test the production build locally: 

# Generate static site and serve it locally
yarn generate:serve

# This will:
# 1. Clear cache
# 2. Run creator scripts
# 3. Generate static HTML files (175+ pages)
# 4. Serve on http://localhost:3000

# Note: You need 'serve' installed globally:
npm install -g serve
Production Build vs Dev Server:

The production build (yarn generate) is different from the dev server (yarn dev):

  • Dev: Hot reload, source maps, faster builds
  • Production: Optimized, minified, static HTML files

Always test with yarn generate:serve before deploying!

3. Netlify Production Deployment

Overview of Netlify Continuous Deployment

ICJIA InfoNet uses Netlify for hosting and continuous deployment. Every push to the main branch automatically triggers a new deployment.

1. Push to main branch
↓
2. Netlify detects change
↓
3. Build environment starts
↓
4. yarn install (dependencies)
↓
5. yarn generate (build)
↓
6. Deploy to CDN
↓
7. Site live! 🎉

Connecting GitHub Repository to Netlify

If you're setting up a new Netlify site or need to reconnect the repository:

  1. Log in to Netlify Dashboard
  2. Click "Add new site" → "Import an existing project"
  3. Choose "GitHub" as the Git provider
  4. Authorize Netlify to access your GitHub account
  5. Select the ICJIA/icjia-infonet-nuxt4 repository
  6. Configure build settings (see below)
  7. Click "Deploy site"

Required Netlify Build Settings

Setting Value Description
Build command yarn generate Runs creator scripts and generates static site
Publish directory dist Output directory containing static files
Production branch main Branch that triggers production deployments
Node version 20 Specified in .nvmrc file (auto-detected)
Netlify Auto-Detects Node Version:

Netlify automatically reads the .nvmrc file to use the correct Node.js version. You don't need to manually configure this in the Netlify dashboard.

Configuring Environment Variables in Netlify

Environment variables must be configured in the Netlify dashboard:

  1. Go to Site settings → Build & deploy → Environment variables
  2. Click "Add a variable"
  3. Add each variable from your .env.sample file
  4. Click "Save"

Required environment variables for production:

NUXT_PUBLIC_API_BASE_URL=https://infonet.icjia-api.cloud
NUXT_PUBLIC_BASE_URL=https://infonet.icjia.illinois.gov
NUXT_MANUAL_ROUTES=["/search", "/translate", "/faqs", "/news","/data-and-publications"]
PLAUSIBLE_DOMAIN=infonet.illinois.gov
PLAUSIBLE_API_HOST=https://analytics.icjia-api.cloud/js/plausible.js
NUXT_BASE_URL=/
NUXT_ACTIVE_SECTIONS=['root', 'page', 'meetings', 'faqs']
Environment Variables Are Sensitive:

Never commit .env files to Git. Environment variables should only be configured in the Netlify dashboard for production.

Build Process Timeline

Phase Duration What Happens
Dependency Installation 1-2 minutes yarn install downloads all packages
Creator Scripts 2-4 minutes Fetch content from Strapi APIs, generate Markdown files
Static Generation 3-5 minutes nuxt generate creates 175+ HTML pages
Asset Optimization 30-60 seconds Netlify optimizes images and assets
CDN Distribution 10-30 seconds Deploy to global CDN
Total 7-12 minutes Complete deployment

Monitoring Deployment Status

To monitor a deployment in progress:

  1. Go to the Deploys tab in Netlify dashboard
  2. Click on the deployment in progress
  3. View real-time build logs
  4. Watch for errors or warnings
Successful Deployment Indicators:
  • ✅ Build status shows "Published"
  • ✅ No errors in build logs
  • ✅ Site preview shows updated content
  • ✅ Production URL loads correctly

Viewing Build Logs and Debugging Failed Deployments

If a deployment fails:

  1. Click on the failed deployment in the Deploys tab
  2. Scroll through the build log to find the error
  3. Common error locations:
    • Dependency installation: Check for version conflicts
    • Creator scripts: Check for API connection issues
    • Nuxt generate: Check for route generation errors
  4. Fix the error locally and push again
Build Failed? Don't Panic!

Failed deployments do NOT affect your live site. The previous successful deployment remains live until a new deployment succeeds.

4. Common Deployment Issues & Solutions

Issue 1: Node.js Version Mismatch

Error Message: 
error unplugin-utils@0.3.1: The engine "node" is incompatible with this module.
Expected version ">=20.19.0". Got "18.20.6"

Cause: Wrong Node.js version installed or Netlify using cached version.

Solution:

# Local fix:
nvm install 20
nvm use 20
node --version  # Verify it shows v20.x.x

# Netlify fix:
# 1. Ensure .nvmrc file contains "20"
# 2. Clear Netlify cache: Site settings → Build & deploy → Clear cache and retry deploy
# 3. Trigger new deployment

Issue 2: Yarn vs npm Conflicts

Error Message: 
Error: Cannot find module 'xyz'
or
Dependency version mismatch

Cause: Mixing npm and Yarn package managers.

Solution:

# Remove npm artifacts
rm -rf node_modules package-lock.json

# Use Yarn exclusively
yarn install

# NEVER run:
npm install  # ❌ Don't do this!

# ALWAYS run:
yarn install  # ✅ Do this!
Remember: This project uses Yarn 1.22.22 exclusively. Using npm will cause conflicts.

Issue 3: Missing Environment Variables

Error Message: 
ERROR: no api base url specified
or
ERROR: no site base url specified

Cause: Environment variables not configured.

Solution:

# Local fix:
# 1. Copy .env.sample to .env
cp .env.sample .env

# 2. Edit .env and add required values
nano .env

# Netlify fix:
# 1. Go to Site settings → Build & deploy → Environment variables
# 2. Add all variables from .env.sample
# 3. Trigger new deployment

Issue 4: Build Script Failures (Creator Scripts)

Error Message: 
Error fetching data from API
or
GraphQL query failed
or
Timeout connecting to Strapi

Cause: Strapi API connection issues or API timeout.

Solution:

# Check API connectivity
curl https://infonet.icjia-api.cloud/graphql

# If API is down, wait and retry
# If API is up but slow, increase timeout in creator scripts

# Local testing:
yarn scripts  # Run creator scripts separately to debug

# Check specific creator script:
node ./creators/createMarkdownPages.js

Issue 5: Cache-Related Issues

Symptoms:
  • Old content still showing after update
  • Build using outdated dependencies
  • Unexpected behavior after code changes

Solution:

# Clear local cache
yarn clear:cache
# This removes .nuxt and .output directories

# Clear Yarn cache (if dependency issues)
yarn cache clean

# Netlify cache clear:
# Site settings → Build & deploy → Clear cache and retry deploy

Issue 6: Memory/Timeout Errors During Nuxt Generate

Error Message: 
JavaScript heap out of memory
or
Build exceeded maximum time limit

Cause: Large number of pages (175+) consuming too much memory.

Solution:

# Increase Node.js memory limit (local)
NODE_OPTIONS="--max-old-space-size=4096" yarn generate

# For Netlify, this is usually not needed as they have sufficient memory
# If it persists, contact Netlify support to increase build limits

Issue 7: Strapi API Connection Failures During Build

Error Message: 
ECONNREFUSED
or
Network request failed
or
API returned 500 error

Cause: Strapi API is down or unreachable.

Solution:

  1. Check Strapi API status: curl https://infonet.icjia-api.cloud
  2. Contact Strapi administrator if API is down
  3. Use local development mode while API is down: yarn dev:local
  4. Retry deployment once API is back online

Issue 8: Static Generation Errors for Specific Routes

Error Message: 
Error generating route: /some/path
or
404 error during prerendering

Cause: Route not included in prerender configuration.

Solution:

# Check nuxt.config.js nitro.prerender.routes
# Ensure all routes are included in appRoutes.json

# Manually add route to NUXT_MANUAL_ROUTES in .env:
NUXT_MANUAL_ROUTES=["/search", "/translate", "/faqs", "/news", "/your-new-route"]

# Regenerate routes:
yarn create:routes

5. Deployment Checklist

Pre-Deployment Checklist

  • Verify Node.js version is 20.x (node --version)
  • Verify Yarn version is 1.22.22 (yarn --version)
  • All dependencies installed (yarn install)
  • Environment variables configured in .env
  • Code changes committed to Git
  • Local development server runs without errors (yarn dev)
  • Production build succeeds locally (yarn generate)
  • Test production build locally (yarn generate:serve)
  • Run accessibility audit (yarn audit:a11y)
  • Check build logs for warnings or errors
  • Verify all creator scripts complete successfully
  • Test critical user paths (navigation, search, content display)
  • Review changes in staging/preview environment

Deployment Steps

  1. Ensure all pre-deployment checks pass
  2. Commit and push changes to main branch
  3. Monitor Netlify deployment in dashboard
  4. Watch build logs for errors
  5. Wait for deployment to complete (7-12 minutes)
  6. Verify deployment status shows "Published"

Post-Deployment Verification

  • Visit production URL and verify site loads
  • Check homepage displays correctly
  • Test navigation between pages
  • Verify search functionality works
  • Check that new content appears correctly
  • Test on mobile devices (responsive design)
  • Verify analytics tracking is working
  • Check browser console for JavaScript errors
  • Test accessibility with screen reader (if major changes)
  • Verify all images load correctly
  • Check that external links work
  • Monitor error logs for 24 hours after deployment

Rollback Procedures (If Deployment Fails)

  1. Go to Netlify dashboard → Deploys tab
  2. Find the last successful deployment
  3. Click the three dots (⋯) next to the deployment
  4. Select "Publish deploy"
  5. Confirm rollback
  6. Site will revert to previous version within 1-2 minutes
  7. Fix the issue locally before attempting new deployment
Rollback is Safe and Fast:

Netlify keeps all previous deployments. You can rollback to any previous version instantly without rebuilding.

6. Tips, Tricks & Best Practices

How to Speed Up Builds

  • Use Netlify's cache: Don't clear cache unless necessary
  • Optimize creator scripts: Run only necessary scripts during development
  • Use local development mode: yarn dev:local skips API calls
  • Parallel processing: Creator scripts run independently when possible
  • Incremental builds: Only push changes when ready to deploy

When to Clear Cache vs When Not To

Clear Cache When... Don't Clear Cache When...
Dependency versions changed Only content changed
Build configuration changed Making small code tweaks
Seeing unexpected behavior Everything is working fine
Node.js version changed Regular development work
Nuxt config changed Just updating content

Testing Production Build Locally

Always test production builds before deploying:

# Full production build and serve
yarn generate:serve

# This simulates exactly what Netlify will deploy
# Test thoroughly at http://localhost:3000

# Check for:
# - All pages load correctly
# - No console errors
# - Images display properly
# - Navigation works
# - Search functionality works

Handling Content Updates from Strapi

When content is updated in Strapi:

  1. Option 1: Automatic (Recommended): Configure Strapi webhook to trigger Netlify build on content changes
  2. Option 2: Manual: Trigger deployment manually in Netlify dashboard
  3. Option 3: Scheduled: Set up scheduled deployments (e.g., daily at 2 AM)
Strapi Webhook Setup:

In Strapi admin panel, go to Settings → Webhooks → Add webhook. Set the URL to your Netlify build hook URL (found in Netlify dashboard under Build & deploy → Build hooks).

Branch Strategy Recommendations

  • main branch: Production-ready code only
  • develop branch: Integration branch for features
  • feature branches: Individual features or fixes
  • Pull requests: Always use PRs for code review
  • Deploy previews: Netlify automatically creates preview URLs for PRs

Using Netlify Deploy Previews for Pull Requests

Netlify automatically creates preview deployments for pull requests:

  1. Create a feature branch and make changes
  2. Push branch to GitHub
  3. Create pull request to main
  4. Netlify automatically builds a preview
  5. Preview URL appears in PR comments
  6. Test changes in preview before merging
  7. Merge PR to deploy to production
Deploy Previews are Powerful:

Use deploy previews to test changes in a production-like environment before merging. Share preview URLs with stakeholders for review.

Performance Optimization Tips

  • Image optimization: Use WebP format and responsive images
  • Code splitting: Nuxt automatically splits code by route
  • Lazy loading: Use lazy loading for images and components
  • CDN caching: Netlify's CDN caches static assets globally
  • Minimize dependencies: Only install necessary packages
  • Monitor bundle size: Keep JavaScript bundles under 250KB

7. Troubleshooting Guide

Deployment Troubleshooting Decision Tree

Start: Deployment Failed

↓ Check build logs in Netlify dashboard

Q: Error during dependency installation?

→ Yes: Check Node.js version, clear cache, verify package.json

→ No: Continue ↓

Q: Error during creator scripts?

→ Yes: Check Strapi API connectivity, verify environment variables

→ No: Continue ↓

Q: Error during nuxt generate?

→ Yes: Check route configuration, verify content files exist

→ No: Continue ↓

Q: Build succeeded but site not working?

→ Yes: Check browser console, verify environment variables

→ No: Contact support
