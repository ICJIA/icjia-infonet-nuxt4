# Accessibility Audit Reports

This directory contains accessibility audit reports generated by axe-core for the Nuxt 4 application.

## Overview

The accessibility audit system automatically scans all pages in the application and identifies WCAG 2.0/2.1 Level A and AA violations, as well as best practice issues.

## Running the Audit

### Prerequisites

1. Make sure the development server is running:
   ```bash
   npm run dev
   ```
   The server should be running at `http://localhost:8000` (default).

2. If your server runs on a different port, set the `AUDIT_BASE_URL` environment variable:
   ```bash
   export AUDIT_BASE_URL=http://localhost:3000
   ```

### Run the Audit

To run a complete accessibility audit on all pages:

```bash
npm run audit:a11y
```

This will:
- Scan all routes defined in `app/data/appRoutes.json`
- Run axe-core accessibility tests on each page
- Generate both JSON and HTML reports
- Display a summary in the console
- Exit with error code 1 if critical or serious issues are found

### View the Report

To open the latest HTML report in your browser:

```bash
npm run audit:a11y:report
```

## Report Files

The audit generates the following files:

- `accessibility-report-latest.html` - The most recent HTML report (human-readable)
- `accessibility-report-latest.json` - The most recent JSON report (machine-readable)
- `accessibility-report-{timestamp}.html` - Timestamped HTML reports
- `accessibility-report-{timestamp}.json` - Timestamped JSON reports

## Understanding the Reports

### Severity Levels

Issues are categorized by severity:

- **🔴 Critical** - Must be fixed immediately. Causes serious barriers for users with disabilities.
- **🟠 Serious** - Should be fixed soon. Causes significant difficulties for users with disabilities.
- **🟡 Moderate** - Should be addressed. May cause some difficulties for users with disabilities.
- **🔵 Minor** - Nice to fix. May cause minor inconveniences for users with disabilities.

### Report Sections

1. **Summary** - Overview of total pages audited and violations by severity
2. **Pages with Most Issues** - Top 20 pages ranked by number of violations
3. **Top Issues Across All Pages** - Most common accessibility problems
4. **Detailed Results by Page** - Complete breakdown of all violations per page

### Violation Details

Each violation includes:
- **Rule ID** - The axe-core rule that failed
- **Description** - What the issue is
- **Impact** - Severity level (critical, serious, moderate, minor)
- **How to fix** - Guidance on resolving the issue
- **Learn more** - Link to detailed documentation
- **Affected elements** - HTML elements that have the issue

## Common Issues and Fixes

### Color Contrast

**Issue**: Text doesn't have sufficient contrast with its background.

**Fix**: Ensure text has a contrast ratio of at least:
- 4.5:1 for normal text
- 3:1 for large text (18pt+ or 14pt+ bold)

### Missing Alt Text

**Issue**: Images don't have alternative text.

**Fix**: Add descriptive `alt` attributes to all `<img>` tags:
```html
<img src="chart.png" alt="Bar chart showing sales data for Q4 2024">
```

### Form Labels

**Issue**: Form inputs don't have associated labels.

**Fix**: Use `<label>` elements or `aria-label` attributes:
```html
<label for="email">Email Address</label>
<input type="email" id="email" name="email">
```

### Heading Order

**Issue**: Heading levels are skipped (e.g., h1 to h3).

**Fix**: Use headings in sequential order (h1, h2, h3, etc.).

### ARIA Attributes

**Issue**: Invalid or improper use of ARIA attributes.

**Fix**: Follow ARIA authoring practices and only use ARIA when necessary.

## Integration with CI/CD

You can integrate the accessibility audit into your CI/CD pipeline:

```yaml
# Example GitHub Actions workflow
- name: Run Accessibility Audit
  run: |
    npm run dev &
    sleep 10
    npm run audit:a11y
```

The audit will exit with code 1 if critical or serious issues are found, failing the CI build.

## Customization

### Changing the Base URL

Set the `AUDIT_BASE_URL` environment variable:

```bash
AUDIT_BASE_URL=https://staging.example.com npm run audit:a11y
```

### Modifying Audit Rules

Edit `scripts/accessibility-audit.mjs` and adjust the `withTags()` configuration:

```javascript
const results = await new AxeBuilder({ page })
  .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa', 'best-practice'])
  .analyze();
```

Available tags:
- `wcag2a`, `wcag2aa` - WCAG 2.0 Level A and AA
- `wcag21a`, `wcag21aa` - WCAG 2.1 Level A and AA
- `wcag22aa` - WCAG 2.2 Level AA
- `best-practice` - Best practice recommendations
- `section508` - Section 508 compliance

## Troubleshooting

### "No accessibility report found"

Make sure you've run the audit at least once:
```bash
npm run audit:a11y
```

### Timeout Errors

If pages are timing out, increase the timeout in `scripts/accessibility-audit.mjs`:
```javascript
const response = await page.goto(url, { 
  waitUntil: 'networkidle',
  timeout: 60000  // Increase from 30000 to 60000
});
```

### Server Not Running

Ensure your development server is running before starting the audit:
```bash
npm run dev
```

## Best Practices

1. **Run audits regularly** - Make accessibility testing part of your development workflow
2. **Fix critical issues first** - Prioritize by severity level
3. **Test with real users** - Automated testing catches ~30-40% of issues; manual testing is essential
4. **Use semantic HTML** - Proper HTML structure prevents many accessibility issues
5. **Test with assistive technology** - Use screen readers and keyboard navigation to verify fixes
